# Thinking Ledger

## Problem

The docs I authored in PR #7 used em dashes, inline colons in prose, and a few banned adverbs. The repo's own anti-slop guidance bans these patterns, and the user invoked `/no-ai-slop` to clean them up.

## Current Evidence

- `grep -nE "—" *.md` found em dashes across ARCHITECTURE.md, CATALOG.md, CLAUDE.md, GEMINI.md, repo-audit.md, AGENTS.md (my sections), and the thinking ledger.
- `grep -nE "(honestly|actually|genuinely)"` found three banned adverbs in prose I wrote.
- Inline colon scan found prose colons introducing inline lists in CONTEXT.md, GEMINI.md, and the thinking ledger.
- The repo's `make slop-scan` does not check em dashes or inline colons because those are not in the banned-phrase regex. The slop scanner targets phrase patterns. The no-ai-slop skill is a complementary manual pass.

## Assumptions

- Cleanup applies only to prose I authored. PR #6's content in AGENTS.md (App design mode, Code quality commands) stays as the user merged it.
- Em dashes in table cells used as "(none)" are still em dashes. Replace with `(none)` for consistency.
- Inline colons in structured glossary entries (`_Avoid_: foo`) are formatting conventions, not the dramatic prose colon the skill targets. Leave those.
- Inline colons in table cells separating category from items are reference structure. Leave those.
- "every check" and "every PR" are concrete coverage statements, not lazy extremes. Leave those.

## Alternatives

- **Add em-dash and inline-colon checks to the slop scanner.** Rejected for this PR. The skill description and the cleanup live as a manual pass for now. Mechanically scanning prose-only colons is hard (the scanner would have to track whether the colon precedes a bullet list, which is brittle in markdown). Revisit if these patterns keep regressing.
- **Rewrite the AGENTS.md PR #6 content as well.** Rejected. The user merged PR #6 deliberately. Editing that content here would erase a recent merge.
- **Aggressively rewrite all "title. elaboration." bullet patterns in CONTEXT.md.** Rejected. The pattern matches recognized technical-writing style for principle lists. Over-correcting would hurt readability.

## Chosen Approach

- Remove every em dash from prose I authored.
- Replace prose inline-colon lists with "like", "such as", "including", or a restructured sentence.
- Cut adverbs (`honestly`, `actually`, `genuinely`) where I had used them.
- Leave PR #6 content, structured glossary `_Avoid_:` entries, and table cell colons alone.
- Verify `make prod-gate` still passes.

## Verification

- `grep -c "—"` across the edited files: 0.
- `make prod-gate`: exit 0. 13 tests passing. Skills graded A. Slop scan ok.
- Diff stat: 9 files, +36 / -36 lines (pure prose substitution).

## Residual Risk

- The slop scanner cannot enforce the em-dash and inline-colon rules from the no-ai-slop skill. Regression is likely without a programmatic check. Future work could add one.
- "Title. Elaboration." patterns in principle lists remain. These are a style judgment, not a clear AI tell.
- Other prose I did not author (PR #6 content, older docs) may still contain the patterns the skill flags.
