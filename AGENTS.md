# AGENTS.md

## Repo contract

This repo is enforced by `make prod-gate`. Read this section first.

Before editing:

1. Read `CONTEXT.md` — mission, principles, anti-slop rules, current assumptions.
2. Read `repo-audit.md` — standing list of duplications, gaps, and risks.
3. Read `ARCHITECTURE.md` — current structure and lifecycles.

Before finishing:

- Update `CATALOG.md` if you added, removed, or renamed a skill / hook / script.
- Update `ARCHITECTURE.md` only when the top-level structure or a lifecycle actually changes.
- Update `skills_index.json` if you added or removed a skill.
- Update `repo-audit.md` if you introduced a new source of truth, a duplication, or a new risk.
- Update `productionization-report.md` if the harness or check set changed.
- Run `make prod-gate`. If it fails, fix the real issue. Do not weaken the gate.

## Purpose

This repository should be cleaned and maintained as a serious senior infrastructure/platform engineering portfolio repo. The code should look like it was written by a practical engineer, not generated by AI.

## Engineering style

- Prefer simple, boring, readable code.
- KISS, DRY, YAGNI.
- Avoid unnecessary abstractions.
- Avoid fake `enterprise-grade` wording.
- Avoid inflated README claims.
- Avoid clever code when clear code works.
- Remove unused code instead of preserving it "just in case."
- Preserve working behavior unless the current behavior is clearly wrong.
- Do not introduce new libraries unless they solve a real problem.

## Agent operating mode

Maximize useful reasoning before editing:

- Build a small mental model of the repo first: purpose, users, entrypoints, data flow, tests, deploy/runtime path, and current failure modes.
- Treat every change as a hypothesis tied to evidence from files, tests, or commands.
- Prefer high-signal fixes: stale docs, broken setup, failing tests, unused dependencies, dead code, misleading examples, security leaks, and behavior gaps.
- Do not perform keyword-only cleanup. Search terms are leads, not proof.
- Do not hide incomplete work with polished language. Either implement the missing behavior, delete the claim, or mark the limitation plainly.
- Preserve domain intent. If the repo is a prototype, make it an honest, runnable prototype rather than pretending it is a platform.
- Keep a running list of risks and unknowns while working.
- Before large edits, identify the smallest safe change that improves the repo's portfolio signal.
- After edits, compare the diff against the repo purpose and revert your own changes if they do not clearly help.
- Prefer tests that lock real behavior over tests that only assert mocks, plumbing, or implementation details.
- Develop autonomously by default. Do not stop for preference questions when a conservative, reversible choice is available.
- Use relevant Codex skills as self-gates before and after edits: repo cleanup, app design, frontend design, security, testing, review, or domain-specific skills when they apply.
- Default autonomous gates are `autonomous-development-loop` for execution and `self-gated-delivery` before final response.
- State assumptions in the final summary instead of asking upfront, unless the missing answer blocks progress or makes the work high-risk.
- Ask only for required permissions, destructive or irreversible actions, secrets/credentials, legal/medical/financial decisions, or product choices that cannot be safely inferred.

## App design mode

When designing or changing an app, maximize intelligence at the product and system level before polishing UI:

- Identify the target user, their job to be done, their context, their constraints, and what success looks like.
- Start from the core workflow and make the first screen useful for that workflow. Do not default to a marketing landing page.
- Model the domain objects, user inputs, system outputs, state transitions, permissions, and audit/history needs.
- Decide what the AI should observe, infer, generate, recommend, ask, refuse, and log.
- Use AI where it reduces cognitive load or improves judgment; use deterministic rules where precision, repeatability, or user control matter more.
- Keep humans in the loop for legal, medical, financial, safety, identity, access, deployment, destructive, or irreversible actions.
- Treat AI output as proposed until accepted when consequences are meaningful.
- Make uncertainty visible when useful: show sources, confidence, assumptions, missing inputs, and ways to correct the model.
- Design correction paths: edit, regenerate, dismiss, explain why, mark wrong, and provide better context.
- Design all important states: empty, loading, partial data, error, conflict, long-running work, success, undo, and degraded AI behavior.
- For operational tools, favor dense, navigable work surfaces over decorative cards: search, filters, lists/tables, detail panes, bulk actions, imports/exports, and activity history when relevant.
- Use realistic sample data and domain language. Avoid generic names, vanity metrics, fake integrations, and unsupported "AI-powered" claims.
- Define validation before expanding scope: acceptance tests, golden workflows, eval fixtures, telemetry, and manual review checkpoints.
- Keep secrets, sensitive prompts, private documents, and personal data out of the repo.
- Gate app work before finalizing: product fit, AI boundaries, UX states, data/security, accessibility, performance, and test coverage.

## Cleanup priorities

When asked to clean or review this repo:

1. Inspect the repo before editing.
2. Identify the real purpose of the project.
3. Remove AI-slop from README, docs, comments, examples, and code.
4. Remove dead code, unused files, commented-out blocks, fake examples, empty tests, and misleading stubs.
5. Search for unfinished-work markers, empty implementations, placeholder copy, filler claims, and fake examples.
6. Implement small missing pieces only when they fit the repo purpose.
7. Delete misleading or unfinished features.
8. Keep documentation factual and runnable.
9. Make setup, test, and usage commands obvious.
10. Add tests only for real behavior.

## Documentation rules

README should explain:

- what the repo does
- why it exists
- how to run it
- how to test it
- one realistic usage example
- known limitations
- next steps only if they are real

README should not include:

- fake production claims
- fake badges
- fake roadmap items
- unsupported architecture diagrams
- marketing fluff
- AI-generated buzzwords like `seamlessly`, `revolutionary`, `cutting-edge`, `world-class`, `best-in-class`

## Code quality commands

Use the commands that apply to this repo. Do not install random tooling unless asked.

### Python

If the repo has a local virtual environment, use it:

```bash
source .venv/bin/activate
python -m pip install -e .  # only if the repo is packaged and this is documented or clearly needed
```

Prefer `.venv/bin/python`, `.venv/bin/ruff`, and `.venv/bin/pytest` when running commands non-interactively.

```bash
python -m compileall .
ruff check .
ruff format --check .
pytest -q
```

If configured:

```bash
mypy .
pyright
bandit -r .
pip-audit
```

### Go

```bash
gofmt -w .
go test ./...
go vet ./...
```

If configured:

```bash
golangci-lint run
staticcheck ./...
```

### JavaScript / TypeScript

```bash
npm install
npm run lint
npm test
npm run build
```

If package manager differs, use the repo's existing lockfile:

```text
pnpm-lock.yaml => pnpm
yarn.lock => yarn
package-lock.json => npm
```

### Docker

```bash
docker build .
```

Only run Docker if the repo has a Dockerfile and the build is expected to work locally.

## Security hygiene

Check for:

- secrets
- tokens
- private hostnames
- hardcoded credentials
- real API keys
- personal data
- `.env` committed by mistake
- generated logs
- local cache files

Replace secrets with environment variables and `.env.example`.
